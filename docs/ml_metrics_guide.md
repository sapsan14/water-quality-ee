## 1. ROC-AUC — умеет ли модель разделять классы?

### Что это

**ROC** = Receiver Operating Characteristic  
**AUC** = Area Under Curve (площадь под кривой)

Модель классификации не просто говорит «да / нет». Она выдаёт **вероятность**:

```
0.01 → чистая вода (модель уверена)
0.50 → модель «не знает»
0.99 → загрязнённая вода (модель уверена)
```

ROC-кривая показывает, **как модель разделяет классы при разных порогах принятия решения**.

### Оси ROC-кривой

| Ось | Что означает | Формула |
|-----|-------------|---------|
| **X** — False Positive Rate | Доля ложных тревог: из всех **чистых** проб — сколько ошибочно назвали грязными | FP / (FP + TN) |
| **Y** — True Positive Rate | Доля пойманных нарушений: из всех **реально грязных** проб — сколько модель нашла | TP / (TP + FN) |

### Как читать AUC

| AUC | Интерпретация |
|-----|---------------|
| 0.50 | Случайное угадывание (монетка) |
| 0.70 | Нормально, модель учится |
| 0.80 | Хорошо |
| 0.90+ | Отлично |
| 1.00 | Идеальное разделение |

### Интуиция

Представьте, что вы берёте **одну случайную грязную** пробу и **одну случайную чистую**. AUC — это вероятность того, что модель присвоит грязной пробе **более высокий** score нарушения, чем чистой.

AUC = 0.99 → в 99% случаев модель правильно ранжирует пары.

### В нашем проекте

ROC-AUC отвечает на вопрос: **насколько хорошо модель вообще отличает чистую воду от загрязнённой**, не привязываясь к конкретному порогу. Это первый уровень оценки — «умеет ли модель?»

**Результаты** (темпоральный split, `06_advanced_models.ipynb`):

| Модель | ROC-AUC |
|--------|---------|
| GradientBoosting | 0.975 |
| **LightGBM** | **0.988** |

## 2. Precision / Recall — какие ошибки делает модель?

### Зачем два показателя

ROC-AUC говорит «модель умеет разделять», но **не говорит, какие ошибки она делает**. А ошибки бывают двух видов:

| Тип ошибки | Название | Что произошло | Опасность для воды |
|------------|----------|---------------|-------------------|
| **False Positive (FP)** | Ложная тревога | Чистую воду назвали грязной | Лишняя проверка лаборатории |
| **False Negative (FN)** | Пропущенное нарушение | Грязную воду назвали чистой | **Люди пьют/купаются в опасной воде!** |

### Precision (точность предсказания)

> Из всего, что модель **назвала «нарушение»** — сколько **реально** нарушений?

```
Precision = TP / (TP + FP)
```

Высокий Precision → модель **не кричит «волки!» зря**.

### Recall (полнота)

> Из всех **реальных нарушений** — сколько модель **нашла**?

```
Recall = TP / (TP + FN)
```

Высокий Recall → модель **ничего не пропускает**.

### Trade-off: нельзя иметь оба идеально

Это фундаментальный компромисс:

```
↑ Recall → ↓ Precision (ловим всё, но и ложных тревог больше)
↑ Precision → ↓ Recall (тревоги точные, но часть нарушений пропускаем)
```

### F1-score: компромисс

```
F1 = 2 × (Precision × Recall) / (Precision + Recall)
```

F1 — гармоническое среднее. Помогает сравнивать модели, когда нужен баланс.

### Почему для воды Recall важнее

В проекте water-quality-ee **приоритетная метрика — Recall класса 0 (нарушений)**.

False Negative = модель предсказала «норма», а вода содержит E. coli → **люди заболели**.

Мы используем **`best_threshold_max_recall_at_precision`** в `src/evaluate.py`: максимизируем Recall при условии Precision ≥ min_precision (обычно 0.70).

### Confusion Matrix (матрица ошибок)

Всё сводится к таблице 2×2:

```
                 Предсказание
                 Нарушение    Норма
Факт  Нарушение    TP          FN  ← пропущенные нарушения!
      Норма        FP          TN
```

В `plot_confusion_matrix()` из `evaluate.py` ячейка FN **обведена красной рамкой** — это самая опасная ошибка.

### Результаты проекта

| Модель (temporal split) | Recall (нарушение) | Precision (нарушение) | F1 |
|------------------------|--------------------|-----------------------|-----|
| GradientBoosting | 0.775 | 0.891 | 0.829 |
| **LightGBM** | **0.956** | **0.881** | **0.917** |

LightGBM пропускает **лишь ~4.4%** реальных нарушений.

## 3. Калибровка — можно ли доверять вероятностям?

### Проблема

Модель говорит: `P(нарушение) = 0.90` — значит ли это, что в 90% случаев с такой оценкой действительно будет нарушение?

**Не обязательно.** Многие модели (включая бустинги) **плохо откалиброваны**: их вероятности — это ранжирующие score, а не настоящие вероятности.

### Пример

| Предсказание модели | Реальная доля нарушений |
|---------------------|------------------------|
| 0.90 | 0.70 |
| 0.80 | 0.60 |
| 0.20 | 0.10 |

→ Модель **переоценивает** уверенность.

### Calibration curve (кривая калибровки)

**Идеал** — диагональ 45°: предсказание = реальность.

```
Предсказанная вероятность нарушения
 1.0 ┤                          ·
     │                       ·
     │                    ·      ← идеал (диагональ)
     │                 ·
     │              ·
     │           ·
     │        ·
     │     ·
     │  ·
 0.0 ├─────────────────────────
     0.0                     1.0
         Реальная доля нарушений
```

Если кривая **выше** диагонали — модель **недооценивает** вероятность.
Если **ниже** — **переоценивает**.

### Почему это важно в проекте

В water-quality-ee мы принимаем решения вида: «при P(нарушение) > 0.3 — отправить пробу на дополнительную проверку». Если калибровка плохая:

- Порог 0.3 может пропускать **реальные нарушения** (модель даёт им P = 0.1)
- Или наоборот — давать **слишком много ложных тревог**

### Как исправляют

| Метод | Суть | Когда лучше |
|-------|------|-------------|
| **Platt scaling** | Логистическая регрессия на предсказаниях | Мало данных, S-образное искажение |
| **Isotonic regression** | Немонотонная пошаговая функция | Много данных, сложные искажения |

В ноутбуке `06_advanced_models.ipynb` используется **`CalibratedClassifierCV(..., method='isotonic', cv=5)`** поверх LightGBM.

### Результат калибровки

| Показатель | До калибровки | После (isotonic) |
|------------|--------------|------------------|
| Средняя P(нарушение) для **реально** нарушающих проб | ~0.045 | ~0.114 |

Калиброванные вероятности ближе к реальной доле нарушений (~7.9%).

## 4. SHAP — почему модель приняла решение?

### Что это

**SHAP** = **SH**apley **A**dditive ex**P**lanations

Основано на **значениях Шепли** из теории кооперативных игр: честное распределение «выигрыша» (предсказания) между «игроками» (признаками).

### Ключевая идея

SHAP отвечает: **«Сколько каждый признак _внёс_ в конкретное предсказание?»**

Пример:

```
Модель предсказала: P(нарушение) = 0.82

Разложение SHAP:
  baseline (средний риск):         0.30

  + iron (железо высокое)        → +0.28
  + coliforms (бактерии)         → +0.18
  + turbidity (мутность)         → +0.12
  − ph (pH в норме)              → −0.06
  = итого                        → 0.82
```

Каждое слагаемое — **вклад конкретного признака** в отклонение от средней вероятности.

### Почему SHAP, а не feature importance

| Feature Importance (деревья) | SHAP |
|-----------------------------|------|
| **Глобальная** — средняя важность по всем примерам | **Глобальная + локальная** (для каждого примера отдельно) |
| Не показывает **направление** влияния | Показывает: `iron > 0.5` → **увеличивает** риск |
| Зависит от метрики (gini, gain) | Теоретически обоснован (аксиомы Шепли) |

### 2 уровня SHAP

#### Глобальный (summary)

Какие признаки **в целом** влияют на решения модели? Средний |SHAP| по всем примерам.

Из нашего проекта (LightGBM, temporal split, ноутбук 06):

| Признак | mean\|SHAP\| | Что это и в какую сторону |
|---------|-------------|--------------------------|
| `iron` | 1.217 | Железо — главный предиктор нарушений водопровода. Высокое → нарушение |
| `color` | 0.751 | Цветность воды. Высокая → нарушение |
| `coliforms` | 0.591 | Колиформные бактерии — маркер фекального загрязнения |
| `ph` | 0.333 | Отклонение pH от нормы |
| `iron_ratio` | 0.308 | Отношение железа к норме 0.2 мг/л (>1 = превышение) |
| `enterococci` | 0.306 | Кишечные энтерококки |
| `turbidity` | 0.261 | Мутность |
| `domain_supluskoha` | 0.255 | Тип места — открытый водоём vs водопровод |
| `e_coli` | 0.217 | E. coli |
| `month` | 0.188 | Сезонность (лето — повышенный риск) |

#### Локальный (waterfall)

Для **одной конкретной пробы** — почему модель считает её нарушением:

```
Проба #1847 (Anne kanal, июль 2024):
  E(f(x)) = −2.41   ← baseline
  + e_coli = 2400    → +3.12  (норма 500, превышение в 4.8 раза!)
  + enterococci = 890 → +1.45
  + month = 7         → +0.31 (лето, повышенный риск)
  − ph = 7.8          → −0.05 (pH в норме — немного снижает риск)
  = f(x) = 2.42       → P(нарушение) = 0.92
```

### Визуализации SHAP в проекте

| Тип | Что показывает | Где в проекте |
|-----|---------------|---------------|
| **Beeswarm plot** | Каждая точка = один пример; цвет = значение признака; положение = SHAP value. Показывает распределение влияния | `06_advanced_models.ipynb`, блок SHAP |
| **Bar plot** | Средний \|SHAP\| по признакам — глобальная важность | `06_advanced_models.ipynb`, блок SHAP |
| **Waterfall** | Объяснение одного конкретного предсказания | Можно добавить вызовом `shap.plots.waterfall(shap_values[i])` |

### Как читать beeswarm plot

```
← уменьшает предсказание    увеличивает предсказание →
                    |
  iron     ●●●●●●●●|●●●●●●●●●●●●●●
                    |        ↑
                    |   красные точки = высокие значения iron
                    |   справа = SHAP > 0 → увеличивает P(нарушение)
                    |
  e_coli   ●●●●●●●●|●●●●●●
                    |
```

Красный справа = **высокое значение признака увеличивает предсказание нарушения**.
Синий слева = **низкое значение уменьшает предсказание**.

## 5. Как всё связано: pipeline оценки

### 4 уровня понимания модели

```
┌─────────────────────────────────────────────────────────────┐
│  Уровень 1: ROC-AUC                                         │
│  Вопрос: «Умеет ли модель ВООБЩЕ разделять классы?»          │
│  Ответ: AUC = 0.988 → да, отлично                           │
├─────────────────────────────────────────────────────────────┤
│  Уровень 2: Precision / Recall                               │
│  Вопрос: «Какие ОШИБКИ она делает?»                          │
│  Ответ: Recall = 0.956 → пропускает 4.4% нарушений          │
│         Precision = 0.881 → 11.9% ложных тревог              │
├─────────────────────────────────────────────────────────────┤
│  Уровень 3: Calibration                                      │
│  Вопрос: «Можно ли ДОВЕРЯТЬ вероятностям?»                   │
│  Ответ: после isotonic — да, ближе к реальности              │
├─────────────────────────────────────────────────────────────┤
│  Уровень 4: SHAP                                             │
│  Вопрос: «ПОЧЕМУ она приняла это решение?»                   │
│  Ответ: iron + coliforms + color — главные драйверы          │
└─────────────────────────────────────────────────────────────┘
```

### Pipeline в production ML-системе

```
1. Обучить модель (LightGBM)
        ↓
2. Оценить разделение → ROC-AUC
        ↓
3. Оценить ошибки → Precision / Recall + Confusion Matrix
        ↓
4. Подобрать порог → best_threshold_max_recall_at_precision()
        ↓
5. Откалибровать вероятности → CalibratedClassifierCV
        ↓
6. Объяснить решения → SHAP (глобально и локально)
        ↓
7. Деплой → citizen-service (карта с вероятностями)
```

### Подбор порога (Threshold Tuning)

По умолчанию sklearn использует порог 0.5: если P(класс) > 0.5 → предсказываем этот класс. Но для задачи безопасности воды **0.5 — плохой порог**:

```
P(нарушение) = 0.35 → модель скажет «норма»
Но реально: нарушение! → False Negative → опасность для здоровья
```

Функция `best_threshold_max_recall_at_precision()` в `src/evaluate.py`:
1. Перебирает все возможные пороги на PR-кривой
2. Находит порог, максимизирующий **Recall (нарушений)** при условии **Precision ≥ min_precision**
3. Если ограничение недостижимо — выбирает порог с лучшим F1

Это **cost-sensitive decision logic**: осознанный выбор компромисса с учётом цены ошибки.

## Словарь терминов

| Термин | Определение |
|--------|------------|
| **ROC-AUC** | Площадь под ROC-кривой; мера способности модели ранжировать классы (0.5 = случай, 1.0 = идеал) |
| **Precision** | Доля истинных нарушений среди всех предсказанных нарушений |
| **Recall** | Доля найденных нарушений среди всех реальных нарушений |
| **F1-score** | Гармоническое среднее Precision и Recall |
| **Confusion Matrix** | Таблица 2×2 «факт vs предсказание» (TP, TN, FP, FN) |
| **False Negative (FN)** | Пропущенное нарушение — самая опасная ошибка в задаче качества воды |
| **False Positive (FP)** | Ложная тревога — модель назвала чистую воду грязной |
| **Calibration** | Соответствие предсказанных вероятностей реальным долям |
| **SHAP value** | Вклад одного признака в конкретное предсказание (основан на значениях Шепли) |
| **Threshold** | Порог принятия решения: при P(нарушение) > threshold → классифицируем как нарушение |
| **Isotonic regression** | Метод калибровки: подгоняет вероятности немонотонной ступенчатой функцией |
| **TreeExplainer** | Быстрый алгоритм расчёта SHAP values для моделей на деревьях (RF, GBM, LightGBM) |
| **Beeswarm plot** | Визуализация SHAP: каждая точка — пример, цвет — значение признака, позиция — SHAP value |
| **Temporal split** | Разбиение по времени: обучаемся на прошлом, тестируем на будущем — честнее случайного split |
| **TimeSeriesSplit** | K-fold кросс-валидация с соблюдением хронологии: каждый следующий fold — более поздние данные |
