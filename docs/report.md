**Карта рисков:** [h2oatlas.ee](https://h2oatlas.ee) · **Код:** [github.com/sapsan14/water-quality-ee](https://github.com/sapsan14/water-quality-ee)

## 1. Введение

Эстонский Департамент здоровья регулярно публикует результаты лабораторных проб воды из купальных мест и водопроводных сетей. Каждая проба содержит измерения микробиологических и химических параметров и оценку «vastab nõuetele» / «ei vasta nõuetele» (соответствует / не соответствует нормам).

**Задача:** построить бинарную классификационную модель, которая по измеренным параметрам предсказывает, нарушает ли проба воды нормативы (`compliant = 0` — нарушение, `compliant = 1` — норма).

**Практическая значимость:** автоматическая ранняя диагностика потенциально опасных проб до завершения полного лабораторного цикла; приоритизация проверок; оценка риска по неполным данным.

**Приоритетная метрика — Recall класса 0 (нарушений).**  
False Negative (FN) = модель предсказала «норма», а вода содержит E. coli или превышение по химии. Это наихудший исход: люди купаются в опасной воде или пьют загрязнённую воду.

### 1.1 Что именно предсказывает модель

Модель — это **вероятностный оценщик риска нарушения санитарных норм** в пространстве лабораторных измерений. Она отвечает на вопрос: *«по набору измеренных параметров, соответствует ли проба шаблону нарушений из исторических данных?»*

Модель **не** предсказывает:
- Физическое «качество воды» или наличие загрязнителей, которые не были измерены
- Причинно-следственные связи между параметрами и загрязнением
- Будущее качество воды (каждая проба — точечный замер)
- Безопасность в широком медицинском смысле (вирусы, токсины за пределами 15 параметров)

Подробный анализ: [`docs/ml_framing.md`](ml_framing.md).

### 2.1 Важный шаг предобработки: дедупликация мест

**Проблема, обнаруженная в данных:** Terviseamet переименовывает места между годовыми XML файлами. Одно и то же физическое место получает разные строки в поле `location`:

| Год | Название в XML |
|---|---|
| 2021 | `Harku järve supluskoht` |
| 2025 | `Harku järve rand` |
| 2022 | `Haaslava küla veevärk` |
| 2026 | `Haaslava küla ühisveevärk` |
| 2021 | `Tootsi Ujumisbassein` |
| 2026 | `Tootsi ujumisbassein` _(только регистр)_ |

Без нормализации `groupby("location")` создаёт ложные дубли: место выглядит как два разных объекта, одному из которых годами не берут проб. В снимке апреля 2026 обнаружено **38 таких пар** (10 купальных мест, 24 водопровода, 4 бассейна).

**Решение:** функция `normalize_location(name, domain)` в `src/data_loader.py`:
- приводит к нижнему регистру
- убирает суффиксы типа объекта: `supluskoht`, `rand`, `veevärk`, `ühisveevärk`, `veevõrk`
- нормализует дефисы, двойные пробелы, точки с запятой

`load_domain()` / `load_all()` автоматически добавляют столбец `location_key`. **При любой агрегации по месту используйте `location_key`, а не сырой `location`.**

## 4. Методология

### 4.1 Инженерия признаков

Из сырых 15 параметров формируются **53 признака** трёх типов:

1. **Исходные значения** (15 параметров): `e_coli`, `ph`, `iron`, ...
2. **Отношение к норме** (14 признаков): `e_coli_ratio = e_coli / 500`, `manganese_ratio = manganese / 0.05` и т.д. Значение > 1 означает нарушение.
3. **Индикаторы пропуска** (15 признаков): `iron_missing`, `ph_missing`, ... — бинарный флаг. Факт отсутствия измерения несёт информацию о типе пробы.
4. **Временные признаки**: `month`, `year`, `is_summer`, `season_*` (one-hot)
5. **Домен**: `domain_supluskoha`, `domain_veevark` (one-hot)

pH кодируется как `ph_deviation` — расстояние от допустимого диапазона [6.0–9.0]: `max(0, 6.0 - pH, pH - 9.0)`.

### 4.2 Обработка пропусков и масштабирование

1. Признаки без ни одного значения в train → 0
2. `SimpleImputer(strategy='median')` — fit только на train
3. `StandardScaler` — fit только на train, transform на test

### 4.3 Разбивка данных

```
train: 80% = 30 875 проб  |  test: 20% = 7 719 проб
stratify=True (сохраняем долю классов)
random_state=42
```

### 4.4 Модели

| Модель | Ключевые параметры |
|--------|-------------------|
| Logistic Regression | `class_weight='balanced'`, `max_iter=2000` |
| Random Forest | `n_estimators=200`, `max_depth=12`, `class_weight='balanced'` |
| **Gradient Boosting** | `n_estimators=200`, `max_depth=4`, `learning_rate=0.1`, `sample_weight` |

## 5. Результаты

### 5.1 Метрики

| Модель | Accuracy | Precision (нарушение) | **Recall (нарушение)** | F1 (нарушение) | ROC-AUC |
|--------|----------|----------------------|----------------------|---------------|---------|
| Logistic Regression | 0.917 | 0.454 | 0.827 | 0.586 | 0.936 |
| Random Forest | 0.992 | 0.964 | 0.925 | 0.944 | 0.992 |
| **Gradient Boosting** | **0.993** | **0.946** | **0.954** | **0.950** | **0.994** |

**Лучшая модель: Gradient Boosting** — наивысший Recall на нарушениях (0.954) при точности 0.946 и ROC-AUC 0.994.

### 5.2 Confusion Matrix (Gradient Boosting, test set = 7 719 проб)

```
                  Pred: Норма   Pred: Нарушение
True: Нарушение       25           523    ← FN=25 (пропущенные нарушения) | TP=523
True: Норма         7141            30    ← TN=7141                       | FP=30 (ложная тревога)
```

- **FN = 25** пропущенных нарушений из 548 (4.6%) — допустимо для данной задачи (Recall = 0.954)
- **FP = 30** ложных тревог — незначительная нагрузка на лабораторию (Precision = 0.946)

Для сравнения: Logistic Regression пропускает **95 нарушений** (17.3%).

### 5.3 Наивный baseline

Модель "всегда говорить норма": Accuracy = 92.9%, но Recall(0) = **0%** — не обнаруживает ни одного нарушения. Наша модель GB превосходит baseline по всем значимым метрикам.

## 6b. Продвинутые модели: LightGBM + темпоральная валидация

### Мотивация

Случайный train/test split скрывает реальную сложность задачи: модель "видит" данные 2025 года во время обучения. На практике всегда предсказываем **будущее** по **прошлому**. Темпоральная валидация честнее.

LightGBM также устраняет необходимость медианной импутации — он обрабатывает `NaN` нативно, что критично при 50–95% пропусков в ряде параметров.

### Сравнение на темпоральном split (train: ранние пробы → test: последние 20% по дате)

| Модель | Recall (нарушение) | Precision | F1 | ROC-AUC |
|--------|--------------------|-----------|-----|---------|
| GradBoost (temporal) | 0.775 | 0.891 | 0.829 | 0.975 |
| **LightGBM (temporal)** | **0.956** | **0.881** | **0.917** | **0.988** |
| LightGBM+Calibrated | 0.948 | 0.892 | 0.919 | 0.988 |

LightGBM на темпоральном split **превосходит** GradBoost по Recall (0.956 vs 0.775) — нативная обработка пропусков устраняет искажение, вносимое медианной импутацией.

### Калибровка вероятностей

Без калибровки LightGBM выдаёт заниженные вероятности нарушений (среднее P ≈ 0.045 для реальных нарушений). После изотонической калибровки: P ≈ 0.114 — ближе к реальной доле 7.9%.

Калибровка важна для систем раннего предупреждения: позволяет выбирать порог алерта осознанно (например, поднять Recall до 0.97+ при Precision ≈ 0.70).

## 6c. Четыре уровня оценки модели

Полная оценка ML-модели в этом проекте включает четыре последовательных уровня:

| Уровень | Вопрос | Метрика / инструмент | Где реализовано |
|---------|--------|---------------------|-----------------|
| 1 | Умеет ли модель вообще разделять классы? | **ROC-AUC** | `evaluate_model()`, ноутбуки 04–06 |
| 2 | Какие ошибки делает? | **Precision / Recall**, Confusion Matrix, F1 | `compare_models()`, `plot_confusion_matrix()` |
| 3 | Можно ли доверять вероятностям? | **Calibration curve**, isotonic regression | `CalibratedClassifierCV` в ноутбуке 06 |
| 4 | Почему модель приняла решение? | **SHAP** (глобальный + локальный) | `TreeExplainer` в ноутбуке 06 |

Дополнительно используется **подбор порога** (`best_threshold_max_recall_at_precision` в `src/evaluate.py`): вместо стандартного 0.5 выбирается порог, максимизирующий Recall нарушений при заданном минимальном Precision. Это — **cost-sensitive decision logic**, отражающий приоритет безопасности: лучше ложная тревога, чем пропущенное загрязнение.

Подробнее: [`docs/ml_metrics_guide.md`](ml_metrics_guide.md) — руководство с интуицией, формулами, примерами и визуальными объяснениями.

## 7. Ограничения

### Ограничения ML-постановки

7. **Целевая переменная — функция входных данных.** `compliant` определяется по тем же параметрам, которые мы подаём модели. Модель частично «переоткрывает» нормативные пороги из `docs/normy.md`. Высокие метрики (ROC-AUC > 0.99 на случайном split) отражают структуру задачи, а не уникальные способности модели. На темпоральном split метрики ожидаемо ниже — нормативы и методологии анализа меняются.

8. **Модель не устанавливает причинность.** SHAP-значения показывают вклад признака в предсказание, а не физическую причину загрязнения. «Высокое железо повышает P(нарушение)» — это корреляция в данных, а не каузальное утверждение.

9. **Модель не может обнаружить то, что не измерено.** Если параметр не измерялся (например, нитраты в пробе из купального места), модель использует только индикатор пропуска `_missing` — косвенный сигнал о типе пробы, а не о загрязнении.

### Ограничения данных

1. **География частично восстановлена.** В opendata XML поле `maakond` часто пустое; для признака `county` используется модуль `county_infer` (локальные переопределения, кэш геокодера). Это **не** точные координаты объектов; для гражданской карты координаты дополнительно уточняются Nominatim / центроидом уезда / приблизительной точкой.

2. **Высокий процент пропусков.** Нитраты, нитриты, хлориды, сульфаты измеряются лишь в части проб veevark. Медианная импутация — грубое приближение; модели с нативными пропусками (LightGBM) предпочтительнее.

3. **`county_encoded` и split.** В ноутбуках **03** и **06** словарь уездов строится **только на train** (`fit_county_mapping` + `encode_categoricals`); в test неизвестные уезды получают код как `unknown`. Для совместимости `build_dataset(df)` без аргументов по-прежнему кодирует уезд на всём df.

4. **Домены Terviseamet в пайплайне.** В `load_all()` по умолчанию объединяются **`supluskoha`**, **`veevark`**, **`basseinid`**, **`joogivesi`** (файлы `joogiveeallika_veeproovid_YYYY.xml`). **`mineraalvesi`**: стабильных годовых XML в opendata не обнаружено — домен в загрузчик не включён.

### Ограничения методологии

5. **Случайный train/test в ноутбуках 03–05.** Там используется стратифицированное разбиение; **честная** оценка «прошлое → будущее» дана в `06_advanced_models.ipynb` (темпоральный split, TimeSeriesSplit, калибровка).

6. **Гражданский сервис — отдельная упрощённая модель.** Снимок `citizen-service` обучает Random Forest на полном объёме для скорости деплоя; это не копия лучшего LightGBM из §6, а сознательный компромисс. Модель в citizen-service — инструмент визуализации рисков, а не система принятия решений.

10. **Нет drift detection в продакшене.** Модель обучена на данных 2021–2026; при смене методологии анализа или нормативов предсказания могут деградировать. Визуально drift виден в EDA (2024 — повышенная доля нарушений), но автоматического мониторинга нет.

7. **Бассейновые нормы — двойное исправление.**
   - *Phase 8:* Исходно в `NORMS` был один плоский словарь без учёта домена. Добавлен `NORMS_POOL` и domain-conditional логика.
   - *Phase 10:* Обнаружено, что `free_chlorine` [0.2, 0.6] и `combined_chlorine` ≤ 0.4 были неверны (реальные: [0.5, 1.5] и ≤ 0.5 по Sotsiaalministri 49/2019). **288 из 339 compliant-проб** бассейнов имели free_chlorine > 0.6 — все были ложными тревогами. Исправление подняло agree_rate аудита с 81.5% до 90.8%.
   - *Phase 11:* Модели переобучены с исправленными нормами; citizen-service snapshot пересобран.

8. **Мониторинговые пробелы, обнаруженные через снимок.** Анализ snapshot.json (апрель 2026) показал:
   - **71 точка** (`supluskoha` + `veevark` + `joogivesi`), где последняя проба зафиксировала **нарушение**, но новых данных нет (некоторые — с 2021 года).
   - **63 из 68 источников** питьевой воды (`joogivesi`) без пробы более 1 года; медиана — 1088 дней (≈ 3 года).
   - **53 купальных места** (`supluskoha`) без пробы более 1 года, в т.ч. пляжи Таллина (Pirita, Haabneeme, Harku järv).
   Модель не заменяет мониторинг, но позволяет **приоритизировать** повторные проверки по сочетанию высокого прогнозируемого риска и давности последней пробы.

## 8. Выводы и следующие шаги

### Выводы

- Задача решаема с высокой точностью: **Random Forest достигает Recall(0) = 0.929** при **AUC = 0.981** на темпоральном split (train ≤2024 / test 2025+) по **69 536 пробам** из 4 доменов.
- Главные предикторы нарушений: **coliform-бактерии**, **pH**, **железо**, **марганец** и **энтерококки**.
- Индикаторы пропуска (`_missing`) — неожиданно ценные признаки: они кодируют тип и цель пробы.
- Ratio-признаки (отношение к норме) улучшают интерпретируемость и производительность модели.
- Лето значимо хуже весны по качеству купальных вод (8.5% vs 1.8% нарушений).
- **Phase 10 открытие:** пуловые нормы хлора были неверны (`free_chlorine` [0.2, 0.6] → [0.5, 1.5] мг/л). Это давало 288 ложных срабатываний в citizen-service. Исправление поднимает agree_rate аудита с 81.5% до 90.8%.
- **3.1% проб (2 164 из 69 536) не воспроизводимы** из опубликованных параметров — оформлен запрос в Terviseamet.
- **XML-парсер не теряет данных** — live parity scan на 160 МБ XML закрыл гипотезу о потере полей навсегда.
