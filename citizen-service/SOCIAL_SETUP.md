## What was added

| Area | Files | What it does |
|------|-------|--------------|
| Deep-link | `frontend/app/lib/url-state.ts`, edits in `Dashboard.tsx` | Selected place is mirrored into `?place=<id>` so every share link points at one location. Back button restores selection. |
| Share buttons | `frontend/app/components/ShareButtons.tsx`, CSS in `globals.css` | FB / LinkedIn / Copy link + native Web Share API on mobile. Mounted in the place detail card. |
| Metadata | `frontend/app/page.tsx` (new `generateMetadata`), `frontend/app/layout.tsx` (defaults + JSON-LD) | Per-place title / description / og:image. Default branded card. `summary_large_image` Twitter card. `WebSite` + `Dataset` JSON-LD. |
| SEO | `frontend/app/sitemap.ts`, `frontend/app/robots.ts`, `frontend/app/manifest.ts` | Top-200 places sitemap, allow-all robots, PWA manifest. |
| Assets | `frontend/public/og-default.png`, `frontend/public/apple-touch-icon.png` | Branded fallback OG image and home-screen icon, generated by `citizen-service/scripts/build_social_assets.py`. |
| OG worker | `og-worker/` (new sub-project) | Cloudflare Worker that renders 1200×630 PNG cards on `og.h2oatlas.ee/og?place=<id>`. |
| Build pipeline | `citizen-service/scripts/export_frontend_snapshot.py` (extended) | Now also emits `frontend/public/data/og-index.json` (~250 KB) — the worker and `generateMetadata()` read this. |
| Autopost | `scripts/social/{select-top,render-text,post-fb,post-linkedin,update-state}.mjs`, `.github/workflows/social-digest.yml`, `state/social-last-post.json` | Weekly GitHub Action picks top-N places, posts to FB + LinkedIn, commits state. Has `--dry-run` mode. |

## Local testing — frontend

```bash
cd frontend
npm install                # if you haven't yet
npm run dev                # http://localhost:3000
```

**Important — fetch overrides for first-time local dev.**
Until the first production deploy populates `https://h2oatlas.ee/data/og-index.json`,
`generateMetadata()` and `sitemap()` would fetch from prod and silently fall back
to empty results. Run dev with these overrides instead:

```bash
OG_INDEX_FETCH_BASE=http://localhost:3000 \
NEXT_PUBLIC_OG_WORKER_URL=http://localhost:8787 \
npm run dev
```

- `OG_INDEX_FETCH_BASE` makes the dev server read its own `/data/og-index.json`
  (already on disk after `python3 citizen-service/scripts/export_frontend_snapshot.py`).
- `NEXT_PUBLIC_OG_WORKER_URL` points the rendered `og:image` URL at your local
  `wrangler dev` instead of `og.h2oatlas.ee`.

**Things to verify by clicking around:**

| Check | How |
|-------|-----|
| `?place=<id>` appears in the URL when you click a pin | Click any marker. URL bar should update without a page reload. |
| Reload preserves selection | After URL contains `?place=…`, hit `Cmd-R`. The same point should be re-selected once the snapshot loads. |
| Back button removes selection | Click a place, then hit Back. The detail panel should clear. |
| Share buttons render in the place card | Open any place. Below the model probabilities you should see Facebook, LinkedIn, Copy link buttons. On a mobile viewport (responsive devtools) the native "Поделиться…" button replaces the grid. |
| Copy link copies the deep URL | Click "Скопировать ссылку" — paste into another tab. Should open the same place. |
| `generateMetadata` produces per-place tags | Open `view-source:http://localhost:3000/?place=<id>` and confirm `<meta property="og:title">`, `og:description`, `og:image`, `og:url`, `<link rel="canonical">` reflect the place. |
| Sitemap renders | Visit `http://localhost:3000/sitemap.xml`. Should include the root URL plus up to 200 `?place=<id>` entries (depending on og-index size). |
| Robots renders | Visit `http://localhost:3000/robots.txt`. Should allow `*` and reference the sitemap. |
| Manifest renders | Visit `http://localhost:3000/manifest.webmanifest`. JSON should validate. |
| JSON-LD is present | Search the source for `application/ld+json`. Should see one block with `WebSite` + `Dataset`. |
| Default OG image loads | Visit `http://localhost:3000/og-default.png`. PNG should render. |

**Sharing intent buttons in dev:** Facebook and LinkedIn require a publicly reachable URL when scraping. For end-to-end OG preview testing on a localhost URL, use either:

- a Cloudflare preview deploy (`git push` → CF Pages auto-builds a preview URL), then test that URL in the [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/) and [LinkedIn Post Inspector](https://www.linkedin.com/post-inspector/), or
- `cloudflared tunnel --url http://localhost:3000` to expose dev temporarily.

## Local testing — OG worker

```bash
cd og-worker
npm install
npm run dev          # http://localhost:8787
```

```bash
# Default brand card
open http://localhost:8787/og

# Per-place card (any id from frontend/public/data/og-index.json)
open "http://localhost:8787/og?place=383133"

# Health
curl -i http://localhost:8787/health
```

By default the worker pulls `og-index.json` from production. To test with
your local file, run the frontend dev server *first* and override:

```bash
# In one shell:
cd frontend && npm run dev   # serves localhost:3000, including /data/og-index.json

# In another (note: --var uses colon, not equals):
cd og-worker
npx wrangler dev --port 8787 --local --var OG_INDEX_URL:http://localhost:3000/data/og-index.json
```

Then verify a per-place card renders the actual location name (not the
default brand card):

```bash
# Pick any id from frontend/public/data/og-index.json
PLACE_ID=$(jq -r '.places | keys | .[0]' frontend/public/data/og-index.json)
curl -sI "http://localhost:8787/og?place=${PLACE_ID}"
curl -s "http://localhost:8787/og?place=${PLACE_ID}" -o /tmp/og.png
open /tmp/og.png   # macOS, or `xdg-open` on Linux
```

**Smoke test:** open the dev URL in a browser. The PNG should render in
under a second. Reload — the response should be cached by the browser
(`cache-control: public, max-age=86400, immutable`).

## Local testing — autoposting (dry-run)

The full pipeline, **without touching any social API**:

```bash
# Pick 2 candidates and pretend-post them in Russian
node scripts/social/select-top.mjs --limit 2 \
  | node scripts/social/post-fb.mjs --dry-run --lang ru

# Same, English to LinkedIn
node scripts/social/select-top.mjs --limit 2 \
  | node scripts/social/post-linkedin.mjs --dry-run --lang en
```

The dry-run prints exactly what would be posted (text + URL) and emits a
fake post-id map on stdout. **No tokens required.**

You can also trigger the full GitHub Action manually with `dry_run=true`:

```
GitHub → Actions → "Social digest" → Run workflow → ✓ "Skip API calls"
```

The job logs will show the rendered post bodies for both platforms.
Switch the toggle off only after FB/LinkedIn tokens are in place
(below).

## Production setup — Facebook Page + token

**One-time, ~15 minutes if you already have a Meta account.**

1. **Create the Page.** Go to [Meta Business Suite → Create Page](https://business.facebook.com/latest/create_page). Category: **Science, Technology & Engineering**. Custom URL: `facebook.com/h2oatlas` (or similar — set under Page Settings → Username).
2. **Create a Meta App.** [developers.facebook.com](https://developers.facebook.com) → My Apps → **Create App** → use case **"Other"** → type **"Business"**. Name: `H2O Atlas autoposter`.
3. **Add products.** In the App dashboard → Add product → enable **"Facebook Login for Business"** and **"Pages API"**.
4. **Get a Page access token.**
   - Open [Graph API Explorer](https://developers.facebook.com/tools/explorer/).
   - Top right: pick your app, then **"Get User Access Token"**.
   - Add permissions: `pages_show_list`, `pages_read_engagement`, `pages_manage_posts`. **Generate Access Token**.
   - In the response, click the blue "ⓘ" next to the token → **"Open in Access Token Tool"** → **"Extend Access Token"**. Copy the long-lived (60-day) user token.
   - Back in Graph API Explorer, change the URL to `me/accounts` and `GET`. The response includes `access_token` next to your Page entry — **this token never expires** as long as it stems from a long-lived user token AND you keep `pages_manage_posts` granted. Copy it.
   - Note the Page's `id`.
5. **App Review.** Posting to **your own** Page as an admin works in Dev Mode — no review required. If you ever need to post on someone else's behalf, submit `pages_manage_posts` for App Review (1–2 weeks).
6. **GitHub secrets.** Repo → Settings → Secrets and variables → Actions:
   - `FB_PAGE_ID` = the numeric Page ID
   - `FB_PAGE_ACCESS_TOKEN` = the never-expiring page token

**Test once locally before trusting the workflow:**

```bash
export FB_PAGE_ID=...
export FB_PAGE_ACCESS_TOKEN=...
echo '[{"id":"383133","location":"Prangli saare veevärk","county":"Harju maakond","domain":"veevark","place_kind":"drinking_water","official_compliant":0,"risk_level":"unknown","model_violation_prob":null,"sample_date":"2026-03-25T00:00:00"}]' \
  | node scripts/social/post-fb.mjs --lang en
```

You should see a real `id` in the output and the post on the Page.

## Production setup — LinkedIn token

**Easiest path: post from your personal profile.** Posting from an
**Organization Page** requires Marketing Developer Platform approval
(4–6 weeks); not worth the wait for v1.

1. **Create a LinkedIn app.** [developer.linkedin.com](https://developer.linkedin.com) → **Create App**. Associate it with any LinkedIn page (your personal page is fine for personal posts).
2. **Auth product.** In the app's Products tab, add **"Sign In with LinkedIn using OpenID Connect"** (instant) and **"Share on LinkedIn"** (instant — gives you `w_member_social`).
3. **OAuth flow once locally.** Use [linkedin-developers/oauth-walkthrough](https://learn.microsoft.com/en-us/linkedin/shared/authentication/authorization-code-flow) or any OIDC client. Scopes: `openid profile email w_member_social`. Redirect URI: any localhost callback you can set in app settings.
   - Exchange the auth code for an access token (60-day expiry).
   - Call `GET https://api.linkedin.com/v2/userinfo` to get your `sub` field — that's your member id.
   - Build the URN: `urn:li:person:<sub>`.
4. **GitHub secrets.**
   - `LINKEDIN_ACCESS_TOKEN` = the access token (60 days)
   - `LINKEDIN_AUTHOR_URN` = `urn:li:person:<sub>`

**Important:** the LinkedIn personal token expires every 60 days. Set a
calendar reminder to rotate it; the autopost workflow will start failing
silently otherwise. Long-term, switch to an Organization Page + Marketing
Developer Platform once approved (the org token can be refreshed
programmatically via refresh tokens).

**Test once locally:**

```bash
export LINKEDIN_ACCESS_TOKEN=...
export LINKEDIN_AUTHOR_URN=urn:li:person:...
echo '[{"id":"383133","location":"Prangli saare veevärk","county":"Harju maakond","domain":"veevark","place_kind":"drinking_water","official_compliant":0,"risk_level":"unknown","model_violation_prob":null,"sample_date":"2026-03-25T00:00:00"}]' \
  | node scripts/social/post-linkedin.mjs --lang en
```

## Production setup — OG worker deployment

```bash
cd og-worker
npm install
npx wrangler login        # opens browser for one-time auth
npx wrangler deploy       # creates h2oatlas-og.<account>.workers.dev
```

After the first deploy:

1. Cloudflare dashboard → Workers & Pages → `h2oatlas-og` → **Settings → Triggers → Custom Domains** → **Add Custom Domain** → `og.h2oatlas.ee`.
2. Cloudflare auto-provisions DNS + SSL.
3. Once `https://og.h2oatlas.ee/health` returns `h2oatlas-og worker ok`, uncomment the `routes` block in `og-worker/wrangler.toml` and `npm run deploy` again to lock the binding into version control.

**Verify with social debuggers:**

- [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/) → paste `https://h2oatlas.ee/?place=383133` → **Scrape Again** → preview should show the per-place card.
- [LinkedIn Post Inspector](https://www.linkedin.com/post-inspector/inspect/https%3A%2F%2Fh2oatlas.ee%2F%3Fplace%3D383133).
- [Google Rich Results Test](https://search.google.com/test/rich-results) → confirm `Dataset` JSON-LD is detected.

## Production verification checklist

After everything's deployed, run through this once:

- [ ] `https://h2oatlas.ee/robots.txt` returns 200 and includes the sitemap line.
- [ ] `https://h2oatlas.ee/sitemap.xml` returns 200 and lists `<= 201` URLs (root + ≤ 200 places).
- [ ] `https://h2oatlas.ee/manifest.webmanifest` returns valid JSON.
- [ ] `https://h2oatlas.ee/og-default.png` returns the branded fallback PNG.
- [ ] `https://og.h2oatlas.ee/og?place=<known-id>` returns a place-specific PNG.
- [ ] `https://og.h2oatlas.ee/og` returns the default brand PNG.
- [ ] FB Sharing Debugger preview matches the OG image (force re-scrape if cached).
- [ ] LinkedIn Post Inspector preview matches.
- [ ] First `Social digest` workflow run with `dry_run=false` produces a real post on both platforms.
- [ ] `state/social-last-post.json` is auto-committed by the workflow with `[skip ci]`.

## Rotation / maintenance

| Token | Expiry | What to do |
|-------|--------|------------|
| `FB_PAGE_ACCESS_TOKEN` | Never (if derived from a long-lived user token) | Re-derive only if the originating user removes the app or revokes scopes. |
| `LINKEDIN_ACCESS_TOKEN` | 60 days | Re-run the OAuth flow before expiry. Calendar reminder recommended. |
| OG worker | Auto-deployed via `wrangler deploy` | No expiry, but redeploy when `og-index.json` schema changes. |

If the `Social digest` workflow starts erroring with HTTP 401 on
LinkedIn, that's almost always token expiry — re-OAuth and update
`LINKEDIN_ACCESS_TOKEN`.
